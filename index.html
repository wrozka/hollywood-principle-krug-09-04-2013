An injection

class Job
 attr_reader :retriever, :cleaner

 def initialize(retriever, cleaner)
   @retriever = retriever
   @cleaner = cleaner
 end

 def run(remote, local)
   strm = retriever.get_file(remote)
   cleaned = cleaner.clean(strm)

   File.open(local, 'w') {|f| f.write(cleaned) }
   local
 end
end
DI introduced * replaceable components * created dependencies need to be injected

Hollywood Principle

Don't call us, we'll call you
What it is about? * it means the code doesn't creates its collaborators, they are past to it * it is about how the code is being used -> Inversion of Control

Inversion of control

Control over what?

Two sides * inversion of execution control, command line vs ui programming * control over dependencies instanization is inverted * service locator - dependencies lookup instead of creation * dependency injection - they are given rather than created * the part about dealing with collaborators is called dependency injection

Dependency Inversion

Abstractions should not depend upon details. Details should depend upon abstractions
What those fancy words mean? * again, that you shouldn't create collaborators by hand * statically typed languages, programing against Interfaces * due to Ruby duck typing, programing against anything that is not a concrete class

Manual Dependency Injection

class Injector
  def ftp_job
    Job.new(FtpFileRetriever.new, FileCleaner.new)
  end

  def http_job
    Job.new(HttpFileRetriever.new, FileCleaner.new)
  end
end
Better, but * a lot of repetition * gets more painful, when classes have more dependencies * however, gives the benefits of DI * can be simplified with a little bit of meta programming

Automated Dependency Injection

class Injector
  include Dependor::AutoInject

  def ftp_job
    inject(Job, file_retriver: FtpFileRetriever.new)
  end

  def http_job
    inject(Job, file_retriver: HttpFileRetriever.new)
  end
end
Automated * automated manual dependency injection * convention over configuration * only the difference needs to be specified * dependencies are read from constructor signature

Dependor

  • dependency injection
  • convention over configuration
  • constructor based injection
  • tried and tested
Previous slides shown how dependor works, a quick recap * it is dependency injection, not a service locator * convention over configuration removes a lot of boilerplate, there are no XML files * constructor based injection ensures, that only valid objects are creates * constructor based injection, clearly shows all the dependencies of an object * it was used in ruby, rails and sinatra projects

Some thoughts

  • shines with at least few collaborators
  • when components need to be replaced
  • works nicely for ruby apps
  • think about entry points
Pros * collaborators, object graphs * when components need to be replaced * works well for ruby apps, gems * always think about entry point, big rails apps, may still have simple entry points * can be used in some areas * fightint rails mindset - angular.js vs ember.js * tight coupling vs dependency injection * Ruby tight coupling is not as tigh as in Java/C#

It is not about testing

Testing * This is a talk about dependency injection, so I'm obliged by the law to mention testing. It is super easy to test code written with DI in mind in isolation. * However in Ruby it is easy to stub on anything, so this point is not so valid in our language.I would still say that tests without stubbing on new nor constants are more readable and easier to maintain. Cons of coupling * when stubbing on constants, you can forgot to stub and integrate by accident
